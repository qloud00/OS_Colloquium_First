Вариант 1

1. ООП — это парадигма программирования, которая накладывает ограничения на косвенную
передачу выполнения, реализуя, как правило, это накладыания через механизм виртуальных функция.

Цель ООП: упростить сопровождение и повторное использование кода.

Основные принципы ООП: инкапсуляция, наследование, полиморфизм, абстракция.


2. Объектно-ориентированная декомпозиция

Объектно-ориентированная декомпозиция — это процесс разделения программы на части, каждая из которых моделирует сущность предметной области.


Основные шаги:

	1. Выделение классов.
	2. Определение их атрибутов и методов.
	3. Определение связей между классами).

3. Дизайн ПО

Дизайн программного обеспечения — это процесс проектирования архитектуры, компонентов, интерфейсов и данных, из которых состоит программная система.

Цель: создать структурированную, масштабируемую и поддерживаемую систему.


Примеры паттернов проектирования:

	1.	Singleton — обеспечивает существование только одного экземпляра класса.
	2.	Builder — отделяет процесс построения сложного объекта от его представления.
	3.	Decorator — динамически добавляет новые функции объекту без изменения его структуры.

4. Магическое число 7 +- 2 Миллера.

Закон Миллера:
Человек способен одновременно удерживать в кратковременной памяти от 5 до 9 элементов информации.

Примеры:

	1.	Количество пунктов в меню интерфейса — не более 7.
	2.	Количество элементов в классе — желательно не более 7.
	3.	Количество компонентов в модуле.
	4.	Длина списка параметров функции — до 7 аргументов.
	5.	Количество вкладок или кнопок на панели управления.
	6.	Количество активных задач, отображаемых пользователю.
	7.	Количество элементов, показанных на одном экране без прокрутки.



5. Пять признаков сложных систем по Гради Бучу

По Гради Бучу, сложные системы обладают следующими признаками:

	1.	Иерархичность — система состоит из подсистем.
	2.	Абстракция — возможность описывать систему на разных уровнях.
	3.	Модульность — система разбивается на относительно независимые части.
	4.	Стабильность форм — устойчивость архитектуры при изменениях.
	5.	Множественность представлений — система может быть описана с разных точек зрения (структурной, функциональной, динамической).

---

Примеры IT-систем:

Пример 1: Операционная система (Windows, Linux)

	Иерархичность:ядро - подсистемы - драйверы  процессы 
	Абстракция: пользователь, системные вызовы, аппаратный уровень
	Модульность: модули ядра, сервисы, компоненты GUI
	Стабильность форм: интерфейс API стабилен при обновлениях
	Множественность представлений	: можно рассматривать как набор процессов, файлов, потоков 

Пример 2: Веб-приложение (YouTube)

	Иерархичность: фронтенд - API - микросервисы - БД                                      |	
	Абстракция:  UI, бизнес-логика, данные                                       |
	Модульность: независимые микросервисы (видео, комментарии, рекомендации)             |
	Стабильность форм: API остаётся стабильным при изменении внутренней логики                 |
	Множественность представлений: представление с точки зрения пользователя, разработчика, администратора |
